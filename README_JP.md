## 🎯 VectHarePlus とは？

VectHarePlus は元の VectHare プロジェクトからフォークされた、SillyTavern 向けの**高度な検索拡張生成（RAG）システム**です。新たに英語、日本語、韓国語、繁体字中国語、簡体字中国語の最適化サポートが追加されました。

私が元の VectHare をフォークしたのは、個人の [MVU Game Maker](https://github.com/KritBlade/MVU_Game_Maker) プロジェクトという巨大スケールを扱うためです。これらのプロジェクトの特徴は：
- **極端なスケール：1 ストーリーあたり 2,000 件以上の返信、1 返信あたり 1,000 字以上。要約検索は 3 秒未満で返ってきます**
- 非英語の言語サポート（日本語、韓国語、繁体字/簡体字中国語）。英語はデフォルトでサポート。
- [MVU Game Maker](https://github.com/KritBlade/MVU_Game_Maker) のすべての機能タグを除去。
- 実際に機能する超長期記憶（2000+ メッセージ）

通常の SillyTavern メモリ拡張は、このような負荷では完全にダウンします。特に [MVU Game Maker](https://github.com/KritBlade/MVU_Game_Maker) が使用する機能タグがストーリー内に大量に存在する場合、それらは記憶検索には役立たずです。だから私は、これらの機能タグをすべて整理しつつ、超大規模で高速なベクトル化を維持できるものが必要でした。

ほとんどのメモリ拡張は 100 メッセージ以下のチャット向けに設計されており、そのスケールでは完璧に動作します。しかしチャットがそれを超えて成長すると、古いメッセージを次第に積極的に要約せざるを得なくなります。最終的には、最近の履歴は完全な詳細で残り、古いものはひどく圧縮されたぼやけたものになってしまう——そして 100+ メッセージ分の生コンテキストはプロンプトや自動作成のロアブックエントリには収まらないので、本当に解決方法はありません。古い記憶は**圧縮しなければならず**、つまり詳細が失われます。

四角い車輪の車はどんなにチューニングしてもこの問題を解決しません。仕事に合った正しい道具が必要です。私が実際に必要としているのは、これらすべての記憶を適切に保存するための専用ベクトルデータベースバックエンドです。

これに取り組むため、VectHarePlus は専用のベクトルデータベースを使用してチャットの**意味あるすべてのイベント**を保存します。最初のメッセージであれ 2,000 番目のメッセージであれ、意味のあるすべてのイベントはデータベースに残り、いつでも SillyTavern が検索できる状態になっています。私が望むのは、10k 以上のメッセージにスケールでき、ラウンドトリップ検索が数秒以内に完了する、SillyTavern 向けのプロダクション品質のメモリベクトルシステムです。

### 解決する問題

- 😩 記憶保存前に [MVU Game Maker](https://github.com/KritBlade/MVU_Game_Maker) が使用するすべての機能タグを除去。
- 🧠 [MVU Game Maker](https://github.com/KritBlade/MVU_Game_Maker) のキャラクター基盤の記憶の上にストーリー基盤の記憶を追加。
- 💸 長い会話が無関係な履歴であなたのトークン予算を圧迫
- ✍️ キャラクターに重要なイベントを思い出させるために手動でコンテキストを編集

**VectHarePlus の解決策：** チャットがどれだけ長くなっても、チャットのすべての意味あるイベントを保存する専用ベクトルデータベースとして **Qdrant** を使用します。追加サービスを動かす準備がまだのユーザーには、**A1 と A2 パスを使う「軽量版」** が SillyTavern 組込のベクトルストア上で動作し、追加のソフトウェアは不要です——フルベクトル DB と多くの機能を小規模で共有します。本格的な長期記憶システムの準備ができたら、Qdrant を使う **A3 パス**にアップグレードしてください。

注意：Qdrant は無料でオープンソースです

## 🧠 仕組み

ベクトル検索は、非常に賢い「検索」機能のようなものです。正確な単語をマッチさせるのではなく、**意味**をマッチさせます——「お腹が空いた」と入力すると「ランチに行こう」というメッセージを見つけられます。*意味*が似ているからです。

VectHarePlus は、互いに連動する 2 つのアイデアを中心に構築されています：

### 1. EventBase — 返信ごとの要約ではなく、ウィンドウから抽出されたイベント

ほとんどのメモリ拡張は、各（または複数の）AI 返信を 1 つのテキストの塊に要約します。理屈上は問題なさそうですが、返信に実際に何が含まれているかを見てみると：

- 100 文の返信には、95 文の埋め草の会話、シーン設定、雑談に埋もれた**5 つの意味あるイベント**（戦闘、発見、約束、アイテム交換、関係性の節目）が含まれているかもしれません
- 別の返信には**0 個のイベント**かもしれません——ただの雑談
- 3 つ目は 1000 字に**1 個のイベント**を詰め込んでいるかもしれません。

返信ごとの要約は、これら 3 つのケースすべてを「返信あたり 1 つの塊」にフラット化します——イベント境界を失い、重要な節目と埋め草を混ぜ、実際に何かが起きたかどうかに関係なく同じデータ形状を生み出します。

**EventBase は最近のメッセージのウィンドウを見て、実際に発生したことに応じて 0、1、または複数の構造化イベントを抽出します。** 各イベントは、豊富なメタデータ（`characters`、`locations`、`items`、`concepts`、`importance`、`DateTime` など）を持つ独自のレコードです——単なる説明文字列ではありません。

Tav が Astarion と 3 人の他の仲間と背景ストーリーノイズの中で 100 文に及ぶ長い買い物旅行をした場合、EventBase はその大量のテキストから 1 つのイベントを抽出するかもしれません：

```
{
  description: "Tav and Astarion shopped for armor in Baldur's Gate.
                Astarion mocked the prices. Tav bought a leather chestpiece for 80gp.",
  importance: 0.6,
  source_window: [msg 142 → 154]
}
```

後で「あの買い物旅行を覚えてる？」と言及すると、VectHarePlus は**そのイベント**を検索します——100 の生の文ではなく、買い物旅行を周囲の無関係な雑談と平均化したぼやけた要約でもありません。

### 2. 専用ベクトル DB がなぜ自然な選択肢なのか

返信ごとの要約しかしないのなら、ベクトルデータベースは実は*必要ありません*。常に同じ形状の返信あたり ~1 つの塊を生み出し、チャットが成長するにつれて再帰的に再要約しているだけです。シンプルなテキストファイルで十分です——だからこそ、多くの古いメモリ拡張は本物の DB を使わなかったのです。

しかし EventBase にコミットすると、状況は一変します：

- 2,000 件の返信 → 潜在的に **1,000～3,000 個の構造化イベント**（複数を抽出する返信もあれば、ゼロの返信もある）
- 各イベントは豊富なフィールドを持つ：characters、locations、items、concepts、keywords、importance、timestamps
- ユーザーが会話の最中に、**意味 + キーワード + メタデータフィルタリング**でリアルタイムに関連する 5～10 個のイベントを見つける必要がある

これはまさに **Qdrant のような専用ベクトルデータベースが設計されているワークロード**です：密ベクトル類似度と疎キーワード検索の両方に加えてメタデータフィルタリング、さらに全コーパスにわたるグローバル BM25 重み付けをサポートする、構造化された多数の小規模レコード。要約のフラットファイルでこれをやろうとすると、線形スキャン、キーワードインデックスなし、メタデータフィルタなし、数百エントリ以上にはスケールしないことを意味します。

EventBase は Qdrant を*強制*しません——A1/A2 軽量パスは SillyTavern 組込の Vectra で動作します。しかしチャットが数百イベントを超えると、Qdrant がこのデータ形状のために実際に作られたストレージレイヤーです。

### 🧠 これが従来のメモリ拡張に勝る理由

既存のほとんどのメモリ拡張は 2 つのアプローチのうちの 1 つを使用します。両方ともチャットが成長するにつれて詳細を失います。理由は以下の通りで、EventBase がどのようにそれを回避するかも示します：

| 側面 | 📝 ローリング要約 <br>*（ほとんどの「メモリ」拡張）* | ✂️ 生チャンキング <br>*（古いベクトル RAG）* | 🧬 EventBase <br>*（VectHarePlus）* |
|---|---|---|---|
| **保存されるもの** | 永遠に成長する 1 つの要約テキスト | 生のチャンクに切られたすべてのメッセージ | メタデータ付きの構造化イベントレコード |
| **メッセージ 100 時点** | ほぼ無傷 | 無傷 | 無傷 |
| **メッセージ 200 時点** | 大きく圧縮——名前、数字、一度きりの詳細が漂流または消失 | トークン予算オーバーフロー——古いチャンクはスコアで剪定または削除 | **無傷**——古いイベントはまだ DB にあり、関連性で浮上 |
| **メッセージ 1,000+ 時点** | 事実上ぼやけ | DB 肥大化；生チャンクは低信号なので検索がうるさくなる | **無傷**——現在のシーンに関連する少数のイベントだけが引き出される |
| **「圧縮」が行うこと** | 要約を再帰的に再要約するため、毎回情報が失われる | なし——しかし統合もなし；生テキストは検索に当たり外れ | 一度きり、意味的——*意味あるイベント*を抽出し埋め草を捨てる。イベント自体の詳細は保持される。 |
| **検索信号** | なし——要約全体が常に注入される | 生テキストへのベクトル類似度（言い換えを捉えるがノイズも） | 豊富なフィールド（`characters`、`items`、`locations`、`concepts`、`keywords` に加えて密な意味）へのベクトル + BM25 ハイブリッド |
| **詳細の行き先** | 圧縮されると永遠に失われる | チャンクがスコア閾値を下回ると失われる | **どこにも行かない**——イベントはベクトル DB に生き続け、関連時に浮上する |
| **注入されるもの** | 動作中の要約全体（毎ターン、毎回） | 意味的に近い数件の生メッセージ | 現在のメッセージに重要なイベントのみ |

**核心的な洞察：** ローリング要約が詳細を失うのは、スペースを確保するために古いコンテンツを*捨てる*からです。生チャンキングが詳細を失うのは、*検索がスケールで壊れる*からです。EventBase はあらゆる意味あるイベントを永久に保持し続け——ベクトル + キーワード検索が、今 AI に見せる価値のある 5～10 個を決定させます。詳細は圧縮されず、**無関係性が濾過されます**。

> 💡 **メッセージの言い回しは、何が検索されるかに大きく影響します。** 検索はあなたの返信のテキストによって駆動されるため、使う言葉が重要です。例えば、*「Mayla、なぜ私が身代金を払ったか覚えてる？」* と *「Mayla、なぜ私が 2000 ドル払ったか覚えてる？」* は非常に異なるイベントを返します——「身代金」はそのストーリーラインに結びついたすべてのイベント（誘拐、交渉、受け渡し）を引き出しますが、「2000 ドル」は主に文字通り 2000 という数字に言及するイベントだけにマッチします。AI に特定のシーンを思い出させたいなら、正確な数字のような付随的な詳細ではなく、そのシーンの**ストーリー的に意味のある言葉**でメッセージを錨をつけてください。とはいえ、私がテストしたところ、A3 パスは DB 内の 1500+ イベントの中で両方の入力でも身代金イベントを見つけることができましたが、A1 と A2 パス（下記のパス説明参照）は 2 番目の入力で失敗しました。正しい錨の言葉を使った最初の入力は、A3 パスでも確実により高い品質の検索結果を得られました。

## 🔍 ハイブリッド検索：A1 vs A2 vs A3 パス

VectHarePlus は最良の結果を見つけるために**2 つの信号**を組み合わせます：

- **信号 1 - ベクトル類似度** — 意味ベース（「お腹空いた」が「ランチに行こう」にマッチ）
- **信号 2 - BM25 キーワードスコア** — 完全な単語マッチ（「Astarion」が「Astarion」にマッチ）

バックエンドと設定に応じて、これらを組み合わせる**3 つのパス**があります：（あなたのブラウザから docker 上の専用ベクトルデータベースまで）

### A1 — Standard バックエンド + BM25
ブラウザがベクトル検索を実行して上位 ~100 個の候補を取得し、その候補に対してのみ BM25 キーワードスコアを計算します。シンプルな加重和：`α × vectorScore + β × bm25Score`。

**トレードオフ：** 低速のコンピュータには高速で軽量ですが、完璧なキーワードマッチが上位 100 のベクトル結果の外にあった場合、見えなくなります。

### A2 — Standard バックエンド + Hybrid（A3 パスを使わないほとんどのユーザーに推奨）
A1 と同じですが、以下が追加されます：
- **RRF（Reciprocal Rank Fusion、逆順位融合）** — 生スコアではなく*位置*で結果を結合
- **デュアル信号ボーナス** — *両方*のリストに現れる結果には最大 +8% のブースト

**例：** 「Astarion drinks blood.」を検索。ベクトル（「吸血鬼/空腹」）*と* BM25（文字通りの「Astarion」+「blood」）の両方でマッチしたイベントは、片方のリストにしかないイベントより上位にランクされます。

**トレードオフ：** より高速なコンピュータブラウザでより良い融合を提供しますが、依然としてベクトル top-K 100 候補プールに制限されます。（まだ上位 100 サンプル）

### A3 — Qdrant ネイティブスパース + サーバーサイド RRF（最高の精度）
両方の検索が**単一の API 呼び出しで Qdrant ベクトルデータベース内で実行**されます。各保存ポイントは 2 つのベクトルを持ちます：密（意味）と疎（キーワード頻度）。Qdrant は**全コーパス**にわたって BM25 重みを計算し（真の IDF、バイアスなし）、ネイティブ RRF で融合します。キーワード側は密検索の上位結果に制限されません——イベントにあなたのクエリ単語が含まれていれば、意味ベクトルが近い一致でなくても対象となります。そして BM25 単語重要度の重みはデータベース内の**すべてのイベント**の統計を使って計算されるため（上位 100 サンプルだけではない）、稀な単語も正しくスコア付けされます。

**例：** 「I cast Fireball at the dragon.」を検索。Qdrant は密インデックス（呪文/攻撃の意味用）と疎インデックス（文字通りの「Fireball」と「dragon」用）を同時に検索し、サーバー側で融合し、1 つのランク付けされたリストを返します。

**トレードオフ：** 最高の精度、規模では最速。ただし、Qdrant ベクトルデータベースを実行する追加の docker が必要です。

| バックエンド設定 | 得られるパス |
|---|---|
| Standard（Vectra - 標準 SillyTavern ベクトルフォーマット）+ BM25 | A1 |
| Standard（Vectra - 標準 SillyTavern ベクトルフォーマット）+ Hybrid | A2 |
| Qdrant（docker 上の専用ベクトル DB） | A3 |

## ✨ 機能

### 🧬 EventBase — LLM 抽出のチャット記憶
LLM がチャットを重要度/最近度/持続性の重み付き構造化イベントに要約します。4 重み再ランカーが何を注入するかを決定します。組込のデュプ抑制により、最近のメッセージに既に見えるイベントを抑制します。

各イベントは生のテキストではなく、構造化されたレコードです。LLM が Astarion の買い物ウィンドウから生成するものの例：

```
event_type:    item_acquired
importance:    6
text:          Tav and Astarion shopped for armor in Baldur's Gate. Astarion mocked the prices.
               Tav bought a leather chestpiece for 80gp.
DateTime:      1492-08-15T14:00:00
cause:         Tav needed better armor before the Gauntlet of Shar expedition
result:        Tav now wears the leather chestpiece; 80gp spent from party funds
characters:    [Tav, Astarion]
locations:     [Baldur's Gate, Sorcerous Sundries district]
factions:      []
items:         [leather chestpiece, 80gp]
concepts:      [armor shopping, party economy]
keywords:      [armor, leather, chestpiece, gold, shopping]
open_threads:  [Gauntlet of Shar preparation]
should_persist: false
```

両方の信号(意味 + キーワード)がこの豊富なフィールドセットに対して動作するため、「armor for the dungeon」のようなクエリは concepts/open_threads 経由でヒットし、「Astarion 80gp」は characters/items/keywords 経由でヒットします。この構造は Qdrant ベクトルデータベースのネイティブであり、他のあらゆるメモリ拡張よりはるかに高いヒット率を実現します。

### 🌏 CJK 言語サポート（日本語、韓国語、繁体字/簡体字中国語）
- Jieba WASM（簡体字 + 繁体字中国語）、TinySegmenter（日本語）、Intl.Segmenter（英語/ラテン/韓国語）
- **言語ごとの専用ストップワードリスト** — 日本語、韓国語、繁体字中国語、簡体字中国語（および英語/ラテン）用に別々にキュレーションされた辞書。ストップワードは、日本語の「の・は・を」、中国語の「的・地・得」、韓国語の「의・은・는・을」のように、どこにでも現れるが検索信号がゼロの埋め草の文法粒子です。キーワードスコアリングの前にこれらを除去することが、CJK コンテンツで BM25 ヒット率を高く保つ鍵です——これがないと、すべてのイベントが共通粒子を含むだけであなたのクエリに「マッチ」してしまい、実際の信号語をかき消します。
- CJK トークナイザーモードは upsert 時に**各 Qdrant コレクションにロック**されます——モードを切り替えると警告モーダルが表示されます

### 🔍 ネイティブスパースベクトルハイブリッド検索（Qdrant）
上記の A3 パス——グローバルに正確な BM25 IDF を持つサーバーサイド RRF。クエリごとに 1 回のラウンドトリップ。

### 📝 保存前要約
ベクトル保存前の必須 LLM 要約。OpenRouter とローカル vLLM 互換エンドポイントをサポート。設定可能なプロンプトテンプレート。

### ⏯️ より良いベクトル化コントロール
停止ボタン、一時停止/再開、実行中の Chrome 再起動でも残るフィンガープリントキャッシュ。

### 🔒 チャットごとのコレクションスコープ
新しいコレクションは現在のチャットに自動で有効化されます。「Active for current chat」チェックボックスがチャットロックを制御します。Database Browser のロックボタンは、ロックが*この*チャット用か別のチャット用かを示します。

### 📡 より賢いステータスインジケーター
Auto-Sync カードは、現在のチャットがベクトル化されているか、いくつイベントが存在するかを表示します。World Info カードはベクトル化されたロアブックを名前で表示します。何かが欠けていれば、両方が正しいベクトル化ツールにリンクします。

### 🗂️ タブインターフェース
設定は **Core**（バックエンド、エンベディング、ハイブリッド）、**EventBase**（チャット履歴/アーカイブチャットファイル .jsonl）、**ChunkBase**（ロアブック/ドキュメント/URL/wiki）、**Action**（診断、開発ツール）に分割されています。

### ⚡ 並列ウィンドウ — ベクトル化の高速化
長いチャットのベクトル化は通常 1 ウィンドウずつ処理します：ウィンドウ 1 を LLM に送る → 待つ → 埋め込む → ウィンドウ 2 を送る → 待つ → 埋め込む → … 2000 メッセージのチャットの場合、これは大量の逐次待機です。

**並列ウィンドウ**スライダー（Vectorize Content の Chunking Strategy セクション）を使うと、最大 **8 つの LLM 抽出 + 埋め込み呼び出しを同時に**起動できます。ウィンドウ 1 が抽出されている間にウィンドウ 2–8 も飛行中、総取込時間が劇的に短縮されます。

| スライダー値 | 動作 |
|---|---|
| **1（安全）** | 一度に 1 ウィンドウ。プロバイダー負荷最小、レートリミットリスクなし、最低速。 |
| **2–4** | 軽度の並列。ほとんどのプロバイダーに良い中間地点。 |
| **5–8（高速）** | 攻撃的並列。レートリミットの高いクラウドプロバイダー（OpenRouter、OpenAI、Cohere）に最適。無料層でレートリミットにかかる可能性。 |

厳しいレートリミットの無料層や単一ローカル GPU の場合は **1** を使用。有料クラウドプロバイダーで 2000 メッセージのチャットを 1 時間ではなく数分で取り込みたい場合は **8** にクランクアップ。

### 🧹 多言語キーワード品質
CJK のより良い 1 文字フィルタリング、高信号の 1 文字 RPG/日常生活/学校用語のためのモード固有の例外。

### 🧹 大規模クリーンアップ
混合バックエンド検索、handle ID フィルタリング、その他の元 VectHare からの拡張周りの数多くのバグ修正。

## 🎭 アクティベーションルール

各コレクションカードにはアクティベーションパネルがあります。優先順位チェーンは：

1. **無効**（一時停止ボタン）→ 決してクエリしない
2. **トリガー** → キーワードが最近のメッセージにマッチ → 有効化
3. **詳細条件** → トリガーが空/マッチしない場合、条件ルールを評価 → 有効化
4. **現在のチャットでアクティブ / キャラクターロック** → 手動の常時オンフォールバック
5. **何もマッチしない** → 有効化しない

条件は感情（Character Expressions スプライト検出経由）、キーワード、メッセージ/ターン数、および組み合わせた AND/OR ルールをサポートします。

> ⚠️ **CJK 注意：** トリガーと感情/キーワード条件は**英語のみ**——キーワード辞書は英語で、regex の `\b` 単語境界は CJK 文字間で発火しません。中国語/日本語/韓国語のストーリーでは、代わりに**「現在のチャットでアクティブ」/ キャラクターロック**を使用してください。Message Count / Turn Count 条件は数値で、どの言語でも問題なく動作します。

## ⏳ 時間減衰

古いコンテンツは低いスコアを得るため、本当に関連性がある場合にのみ浮上します。

```
relevance = original_score × (0.5 ^ (message_age / half_life))
```

half-life = 50 の場合：50 メッセージ前のメッセージは 50% の関連性、100 前は 25%、150 前は 12.5%。フロア（デフォルト 0.3）が完全な忘却を防ぎます。重要なチャンクを**時間に対して盲目**としてマークすると免疫になります。A3 パスはこの機能を必要としないはずです。A3 パスを使う本来の目的は、バックエンドがパフォーマンスペナルティなしで完全なデータベースを検索できるようにすることだからです。

> **EventBase 注意：** EventBase は 4 重み再ランカーに独自の最近性ボーナスを組み込んでいます。スタンドアロンの減衰設定は非チャットコンテンツ（ロアブック、ドキュメント）にのみ影響します。


## ❓ よくある質問

**なぜ内部に 2 つの別々のパイプラインがあるのですか？**
内部的に、VectHarePlus はコンテンツタイプによって 2 つの検索パスのいずれかにルーティングします：

| パイプライン | 処理するもの |
|---|---|
| **EventBase** | チャット履歴（ライブチャット + アップロードされた `.jsonl` アーカイブ） |
| **Standard（Chunk）** | 他のすべて：ロアブック、キャラクターカード、URL、ドキュメント、wiki ページ、YouTube トランスクリプト |

互いのコンテンツを見ることは決してありません——だから同じチャットメッセージが 2 回検索される（イベントとして 1 回、生チャンクとして 1 回）ことはありません。通常は気にする必要はなく、単に EventBase がチャットを所有し、標準チャンクパイプラインが他のすべてを所有することを意味します。

**チャット中に CJK トークナイザーモードを変更できますか？**
いいえ——やめてください。CJK トークナイザーモードはセンチネルポイントを介して upsert 時に**各 Qdrant コレクションにロック**されます。すでにコンテンツをベクトル化した後にモードを切り替えると、次のクエリで「Tokenizer mismatch」警告モーダルがトリガーされ、実際の選択肢は次のみです：
1. 元のモードに**戻す**（既存のベクトルを保持）、または
2. 新しいモードで**コレクションを最初から再ベクトル化**（抽出されたすべてのイベントを捨ててやり直し）。

チャットをベクトル化*する前に*トークナイザーモードを選び、それを維持してください。インプレース移行はありません——スパースベクトルは元のモードでトークン化されているため、異なるトークナイザーの出力とは互換性がありません。チャットをベクトル化*する前に*エンベディングモデルも選んで維持してください。

**なぜ EventBase は私の時間減衰設定を無視するのですか？**
EventBase は 4 重み再ランカーに独自の最近性ボーナスが組み込まれています。スタンドアロンの時間減衰設定は非チャットコンテンツ（ロアブック、ドキュメント）にのみ適用されます。これは意図的です——両方を適用するとチャットイベントが二重に減衰します。

**中国語/日本語/韓国語のチャットでトリガー/感情条件を使用できますか？**
信頼できません。キーワード辞書は英語のみで、regex の `\b` 単語境界は CJK 文字間で発火しません。「Active for current chat」/ キャラクターロック、または数値の Message Count / Turn Count 条件を使用してください。

**なぜ「scene」設定が見つからないのですか？元の VectHare にはありました**
Scene サポートは削除されました（チャンクベースのチャット時代の機能で、チャットは現在 EventBase を通って実行されます）。イベントをグループ化することは論理的ではないため、機能は削除されました。

## 🐛 トラブルシューティング

**"No embeddings available"** — メイン ST 設定で Vectors 拡張を有効化、エンベディングプロバイダーを選択、必要に応じて API キーを追加、Diagnostics を実行。

**イベント/チャンクが検索されない** — Vectorize をクリックしてインデックスを作成、スコア閾値を下げる（0.3 を試す）、コレクションが「Active for current chat」または一致するトリガーを持つことを確認。

**"Backend health check failed"** — Qdrant では、Qdrant サーバーが実行中で Similharity プラグインがインストールされていることを確認してください。

**遅いパフォーマンス** — Qdrant + A3 に切り替え（単一ラウンドトリップ、サーバーサイド融合）。EventBase Top K を減らす。ローカル GPU（順次）の代わりに API エンベディングプロバイダー（並列）を使用。

**メモリが重要な詳細を忘れる** — 重要なチャンクを時間に対して盲目としてマーク、減衰フロアを上げる、トリガーキーワードを追加（英語コンテンツのみ）。
